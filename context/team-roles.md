## 架構重要 Context

### 冇 Marketing / Account Servicing 同事
公司而家冇 Marketing 同 Account Servicing 嘅同事（已離職後一直空缺）。影響：
- **Quotation** — 全部由 Ki 親手做
- **Master Job List 維護** — 由 Kary + Sohling 抽時間人手 copy（唔 cost-effective）
- **Client 溝通** — 冇人做 bridge，導演直接同 client 溝通

### 冇獨立 Producer 角色
DOF 冇 Producer，導演要自己做 PM + directing + client management。呢個係結構性 overload。

### Turnover 偏高
好多同事做一兩年就走。影響：
- Internal tool 嘅 onboarding 要簡單、新人要快上手
- 文件化更加重要，唔好 rely on 口頭傳承

### Kay 離開後嘅潛在影響
- 公司會缺少 Graphic Designer
- KV / Poster / Banner 需要由 Motion Graphics 同事分攤，或者外判，或者用 AI generation 補位

## 合作模式

### 導演 ↔ 後期
1. 導演建立 Video Flow working document（Google Doc）
2. 後期同事對住做，各自分工（Editor 做剪接、MoGraph 做字卡同 graphics）
3. 做完喺 Google Doc mark done
4. 導演 QC 成品
5. Client feedback → 導演整理 → update working document（見 `client-feedback-workflow.md`）

### 老闆 ↔ 導演
- 老闆收到 enquiry → assign 導演跟 project
- 導演接 project 時通常只有口頭 briefing（正努力透過 Project Portal 改善）
- 導演負責前期製作直至最終交付

### 後期組內部
- Sohling 做統籌——根據 Calendar 排期安排邊個做邊條片
- Sohling 最需要 Calendar 資訊準確完整，因為佢要用嚟 plan workload

## Working Style Notes（影響 Mugi 嘅認知）

### Calendar 紀律唔一致
- Kary + Sohling 會仔細 mark 所有 milestone 並 update 改動
- 有導演可能**只標記 shooting**，唔標記其他 milestone
- 有導演改咗期**唔 update calendar**
- 結果：Calendar 上嘅資訊**未必 100% 反映現實**——Mugi 搵到嘅 events 可能有過時嘅
- Mugi 應該：如果 Calendar 資訊同用戶講嘅有出入，以**用戶講嘅為準**，協助更新 Calendar

### 文件化程度唔一致
- 唔係每個導演都有建立 Video Flow working document
- 後期同事有時要靠口頭或者自己揣摩
- 呢個唔係 Mugi 可以解決嘅問題，但 Mugi 需要知道：問後期相關問題時，**唔好假設每個 project 都有完整嘅 Google Doc**
